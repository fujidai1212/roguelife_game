# CLAUDE.md — プロジェクト指示書

このファイルは Claude Code への恒常的な指示書である。すべての作業でこの内容に従うこと。

## プロジェクト概要

ダークファンタジー系のテキストベース・ローグライクRPG（スマホアプリ）を開発する。
2015年の名作スマホゲーム『異世界に生きる』『異世界の闇の中で』にインスパイアされたオリジナル作品。

- 詳細なゲーム仕様: `docs/GAME_DESIGN.md`（必読。仕様の唯一の正とする）
- 開発フェーズ計画: `docs/ROADMAP.md`（この順序で実装する。フェーズを飛ばさない）

## 発注者（ユーザー）について

- ゲーム開発・アプリ開発の**完全な初心者**である。
- 専門用語を使うときは短い説明を添えること。
- 各フェーズの完了時に「何ができたか」「どうやって動作確認するか」を日本語で簡潔に報告すること。
- 動作確認の手順（コマンド、スマホでの確認方法）は毎回具体的に書くこと。

## 技術スタック（変更しない）

- **Expo (React Native) + TypeScript**。Expo の managed workflow を使う。
- 対象プラットフォームは **iOS / Android のスマホのみ**。
- 状態管理: Zustand（軽量で初心者にも読みやすいため）
- ローカル保存: AsyncStorage（メタ進行・設定・セーブデータ）
- 乱数: シード可能な PRNG を自作または軽量ライブラリで導入（テストとバランス検証のため）
- テスト: Vitest または Jest（ゲームロジックのユニットテストのみ必須。UIテストは不要）
- 広告・課金は後半フェーズまで**実装しない**（ROADMAP参照）

## アーキテクチャの原則（最重要）

1. **ゲームロジックとUIを完全に分離する。**
   - `src/core/` … 純粋な TypeScript のみ。React / React Native / Expo に一切依存しない。
     戦闘計算、日数・年齢進行、乱数、イベント解決、セーブデータ構造はすべてここに置く。
   - `src/ui/` … 画面・コンポーネント。core の関数を呼ぶだけにする。
   - `src/data/` … 職業・敵・アイテム・イベントなどのマスターデータ（TypeScriptの定数 or JSON）。
2. **ゲーム内のすべての進行は「アクション → 結果」の純粋関数で表現する。**
   `applyAction(state, action) => { newState, log[] }` の形を基本とする。
   これによりテスト・バランス検証・将来のリプレイ機能が容易になる。
3. **マスターデータ駆動。** 敵・イベント・職業・アイテムの追加がデータファイルの追記だけで
   済むように設計する。ハードコードしない。
4. **数値バランスは `src/data/balance.ts` に集約する。** マジックナンバー禁止。

## コーディング規約

- コード（変数名・関数名・コメント以外）は英語。ゲーム内テキスト（プレイヤーに見える文字列）は日本語。
- ゲーム内テキストは `src/data/texts/` に分離し、コンポーネント内に直書きしない（将来の多言語化のため）。
- 1ファイル300行を超えたら分割を検討する。
- `any` 禁止。ゲーム状態・アクション・イベントはすべて型定義する。
- コミットは1機能1コミット。コミットメッセージは日本語でよい。

## ゲーム内テキストのトーン（重要）

- **ダークで乾いた、突き放すような文体**。プレイヤーを「お前」と呼ぶ。
  例: 「お前は目を覚ました。ここがどこかは分からない。分かるのは、金がないことだけだ。」
- ただし**固有名詞は一般的な異世界ファンタジー（なろう系ラノベ・アニメ）の語彙**を使う。
  使ってよい語: ギルド、冒険者、クエスト、ダンジョン、ポーション、スライム、ゴブリン、
  魔石、酒場、受付嬢、薬草、帰還の巻物 など。
  避ける語: 元ネタ2作品に固有の名詞（「闇の欠片」「闇の底」など）。オマージュであって
  コピーではないことを明確にする。
- 死・理不尽・暴力の描写はテキストで淡々と行う。グロテスクな詳細描写・性的表現はしない
  （ストアレーティングを 12+ 相当に収めるため）。

## 画像アセットについて

- ゲーム内画像（敵、背景、UI装飾）は**ユーザーが Gemini で生成して支給する**。
- Claude Code は画像を生成しない。代わりに:
  1. 必要な画像の一覧を `docs/ASSET_LIST.md` に管理する（ファイル名、用途、推奨サイズ、
     Gemini に渡すプロンプト案を日本語で記載）。
  2. 画像が未支給の間は、単色＋テキストのプレースホルダーで動くようにする。
  3. `assets/images/` に規定のファイル名で置けば自動で差し替わる仕組みにする。
- 基調はモノクロ〜ロートーン。UI はほぼ黒背景＋白文字を基本とする。
- **画風の統一（重要）**: Gemini は呼ぶたびに絵柄がブレやすい。`docs/ASSET_LIST.md` の
  冒頭に「共通スタイル指定文」を1つ定義し、すべての画像プロンプト案は必ず
  「共通スタイル指定文 + その画像固有の内容」の形で記載すること。
  共通スタイル指定文の初期案:
  「モノクロの銅版画風イラスト、ダークファンタジー、荒く乾いたタッチ、背景は完全な黒、
  装飾なし、文字なし」
  ※初期案はユーザーが実際に数枚生成して確認後、確定させる。確定後は全プロンプトで固定。
- **透過PNGを前提にしない**: Gemini は背景透過画像を安定して出力できないことがある。
  本作のUIは黒基調のため、すべての画像は「黒背景で生成された非透過PNG」を前提に
  レイアウトを設計すること（画像の縁が黒background に溶け込む前提でよい）。
- **サイズの正規化**: 支給画像のサイズが規定と異なる場合に備え、`assets/images/` 内の
  画像を一括で規定サイズにリサイズ・変換するスクリプト（`scripts/normalize-images.ts`
  等）を用意すること。ユーザーはこのスクリプトの実行方法を知らない前提で、実行手順を
  ASSET_LIST.md に記載すること。
## 作業の進め方

- `docs/ROADMAP.md` のフェーズ順に実装する。**現在のフェーズ外の機能を先取りしない。**
- 各フェーズの開始時に、そのフェーズでやることを箇条書きで宣言してから着手する。
- 各フェーズの完了条件（ROADMAPに記載）をすべて満たしてから次へ進む。
- 仕様に曖昧な点・矛盾を見つけたら、勝手に決めずにユーザーに質問する。
  ただし数値バランスの初期値は仮置きしてよい（`balance.ts` に TODO コメントを残す）。
- ゲームロジック（core）には必ずユニットテストを書く。特に: 戦闘計算、年齢による能力減衰、
  死亡時の魂精算、ダンジョンの深度スケーリング。
